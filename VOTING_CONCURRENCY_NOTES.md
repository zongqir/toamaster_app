# 投票并发处理说明

## 已修复的问题

### 1. ✅ 投票更新原子性问题
**问题描述**: `updateVote()` 原先使用"先删除后插入"的方式，不是原子操作，可能导致数据丢失。

**解决方案**: 
- 创建了数据库RPC函数 `update_vote_atomic()`
- 在数据库事务中完成删除和插入操作
- 确保操作的原子性，失败时自动回滚

**相关文件**:
- `supabase/migrations/00009_add_atomic_update_vote_function.sql`
- `src/db/votingDatabase.ts` (updateVote 方法)

### 2. ✅ 投票结果实时性问题
**问题描述**: 投票结果页面只在进入时加载一次，多人同时投票时看到的是过时数据。

**解决方案**:
- 添加自动刷新机制（每5秒刷新一次）
- 投票关闭后自动停止刷新
- 添加手动刷新按钮
- 添加刷新状态指示器

**相关文件**:
- `src/pages/vote-result/index.tsx`

### 3. ✅ meeting_id 字段缺失
**问题描述**: `updateVote()` 插入新投票时缺少 `meeting_id` 字段。

**解决方案**: 在 RPC 函数中确保插入时包含 `meeting_id` 字段。

### 4. ✅ 计时器后台被杀问题
**问题描述**: 小程序切到后台时，`setInterval` 会暂停，回到前台后时间不准确。

**解决方案**:
- 改用时间戳差值计算：记录开始时间戳 `runStartRef`，每次 tick 用 `Date.now() - runStartRef` 计算真实经过时间
- 将刷新间隔从 1000ms 改为 500ms，后台恢复时更快纠正
- 不需要额外处理 onHide/onShow，时间戳本身是绝对的，不受后台暂停影响

**相关文件**:
- `src/hooks/useMeetingTimer.ts`

**原理说明**:
```typescript
// 之前：每秒 elapsed + 1，后台暂停就停了
setInterval(() => setElapsed(prev => prev + 1), 1000)

// 现在：用时间戳算差值，后台回来自动跳到正确时间
const baseElapsed = 10 // 暂停时已经用了10秒
const runStart = Date.now() // 记录开始时间戳
setInterval(() => {
  const actualElapsed = baseElapsed + Math.floor((Date.now() - runStart) / 1000)
  setElapsed(actualElapsed)
}, 500)
```

## 数据库层面的保护

### 唯一约束
数据库已有唯一约束保护：
```sql
UNIQUE(voting_session_id, voting_group_id, voter_fingerprint, candidate_id)
```

这确保了：
- 同一设备在同一分组中不能对同一候选人重复投票
- 即使前端有竞态条件，数据库也会拒绝重复数据

## 已知限制

### 1. 设备指纹的可靠性
**当前实现**:
- 使用 `navigator.userAgent + 时间戳 + 随机数` 生成指纹
- 存储在本地 Storage 中

**限制**:
- 微信小程序中 `userAgent` 可能为 `undefined`
- 用户清除小程序数据后指纹会重新生成
- 换设备可以重新投票

**适用场景**: 
对于 Toastmasters 俱乐部内部投票，这个精度已足够。成员不太会故意刷票。

### 2. RLS 策略宽松
**当前状态**: 
所有 RLS 策略都是 `USING (true)` / `WITH CHECK (true)`

**影响**:
- 任何人都可以创建/删除投票会话
- 前端使用密码弹窗保护删除操作
- 但直接调用 Supabase API 可以绕过前端校验

**适用场景**: 
俱乐部内部使用，成员之间相互信任。如需更严格的权限控制，需要：
- 实现用户登录系统
- 基于用户角色设置 RLS 策略
- 限制只有会议创建者或管理员可以删除投票

### 3. 投票防刷机制
**当前机制**: 仅依赖设备指纹

**可能的改进方向**（如有需要）:
- 结合微信 OpenID（需要用户授权）
- 添加投票间隔限制
- 记录投票时间，检测异常快速投票
- 添加验证码或人机验证

## 性能考虑

### 自动刷新频率
- 当前设置: 5秒刷新一次
- 适用场景: 10-30人的小型会议
- 如果参会人数更多，可以调整刷新间隔

### 数据库查询优化
- 投票结果查询已基于 `meeting_id` 索引
- 使用了适当的 JOIN 和聚合查询
- 对于大型会议（100+人），可能需要添加缓存层
